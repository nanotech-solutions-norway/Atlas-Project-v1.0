# Atlas Interaction and Module Model

This model adapts the SIW and MIW decision-workspace logic to Atlas.
It preserves the internal sequence:

**Orient → Explain → Fit → Prove → Act**

while staying commercially appropriate for a ChatGPT-first public website.

## 1. Orient
Primary objective: frame the visitor’s context quickly.

Launch modules:
- Outcome selector / use-case selector
- Segment selector
- Package-intent prompt
- Persistent context rail

Captured launch-safe state:
- use case / need
- buyer segment
- package intent
- team size band
- governance sensitivity

## 2. Explain
Primary objective: make the offer legible without technical noise.

Launch modules:
- How It Works page
- Atlas Knowledge blocks
- FAQs
- Bounded explanatory cards on platform fit and package structure

Rule: explain operational outcomes, onboarding, and governance support.
Do not imply platform resale or unbuilt secure runtime.

## 3. Fit
Primary objective: help the visitor identify the right launch-safe package or route.

Launch modules:
- package compare cards
- role-based package guidance
- page modules that switch CTA and proof adjacency based on context

Rule: bind package recommendations to CMS objects, not static page copy.

## 4. Prove
Primary objective: keep trust close to the decision point.

Launch modules:
- proof stack strip
- Trust / Compliance page
- FAQ proof adjacency
- proof cards attached to package pages

Rule: proof must be attached to approved claims and public-safe boundaries.

## 5. Act
Primary objective: move the visitor into the right next step.

Launch modules:
- Request Demo
- Start Onboarding
- Book Consultation
- structured multi-path forms

Routing rule:
CTA labels and targets must be resolved from governed route logic using:
- selected segment
- package intent
- governance sensitivity
- selected platform
- published offer availability

## Global module behavior

### Persistent Context Rail
Carries launch-safe state across pages.
Desktop: sticky side rail or inline block.
Mobile: bottom action dock or collapsible sheet.

### Bounded Demo Surface
The demo must not be an unrestricted open playground.
It should demonstrate value without exposing the reusable operating recipe.

### Future Azure seam
The module model preserves hooks for:
- secure delivery entitlements
- protected downloads
- managed workspace links
- secure-mode CTA routing

These remain unpublished and non-operative at launch.
